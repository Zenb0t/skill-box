# Evidence Bundle Format

## Purpose

An evidence bundle is a structured record of what was verified, what passed, what failed, and what was skipped. It serves as proof of verification for PR reviewers and future audits.

## Standard Format

```markdown
## Verification Evidence

### Context
- **Change**: <1-line description of the change>
- **Type**: <Refactor | Feature | Bug fix | Dependency | Config | Docs | Security | Performance | New module>
- **Level**: <Low | Medium | High | Critical>
- **Branch**: `<branch-name>`
- **Files changed**: <count>
- **Date**: <ISO 8601>

### Checks Performed

| # | Check | Command | Result | Duration | Notes |
|---|-------|---------|--------|----------|-------|
| 1 | Lint | `pnpm lint` | ✓ Pass | 3.2s | No warnings |
| 2 | Typecheck | `pnpm typecheck` | ✓ Pass | 8.1s | |
| 3 | Unit tests | `pnpm test` | ✓ 47/47 passed | 12.3s | |
| 4 | Build | `pnpm build` | ✓ Pass | 15.7s | |

### Gaps

Checks that were NOT performed and why:

- **E2E tests**: Not run — no E2E script configured
- **Performance benchmarks**: Not applicable — not a performance change
- **Security review**: Not applicable — no auth/crypto changes

### Confidence Assessment

**Confidence**: <High | Medium | Low>

**Justification**: <Why this confidence level>

Examples:
- **High**: "Full test suite passed, build succeeds, change is well-scoped refactor"
- **Medium**: "Tests pass but no E2E coverage for the affected user flow"
- **Low**: "Type checks pass but no tests exist for this module"

### Production Checklist

For High/Critical changes only:

- [ ] Database migrations reviewed (if applicable)
- [ ] Environment variables documented (if new ones added)
- [ ] Feature flags configured (if applicable)
- [ ] Rollback plan considered
- [ ] Monitoring/alerting updated (if applicable)
```

## Compact Format (for PR comments)

When embedding in a PR description, use the compact format:

```markdown
## Verification

**Type**: Feature | **Level**: Medium | **Confidence**: High

| Check | Result | Duration |
|-------|--------|----------|
| Lint | ✓ | 3.2s |
| Typecheck | ✓ | 8.1s |
| Tests | ✓ 47/47 | 12.3s |

**Gaps**: No E2E tests configured
```

## Failure Format

When verification fails, use this format:

```markdown
## Verification Evidence

### Context
<same as above>

### Checks Performed

| # | Check | Command | Result | Duration | Notes |
|---|-------|---------|--------|----------|-------|
| 1 | Lint | `pnpm lint` | ✓ Pass | 3.2s | |
| 2 | Typecheck | `pnpm typecheck` | ✗ FAIL | 5.1s | 3 errors |
| 3 | Unit tests | — | ⏭ Skipped | — | Blocked by typecheck |

### Failures

#### Typecheck (3 errors)

```
src/auth/login.ts:42:5 - error TS2345: Argument of type 'string' is not assignable...
src/auth/login.ts:58:12 - error TS2339: Property 'userId' does not exist...
src/utils/helpers.ts:15:3 - error TS7006: Parameter 'x' implicitly has an 'any' type
```

### Suggested Actions
1. Fix type errors in `src/auth/login.ts` (2 errors)
2. Fix type error in `src/utils/helpers.ts` (1 error)
3. Re-run verification after fixes

### Confidence Assessment
**Confidence**: N/A — Verification did not pass
```

## Evidence Storage

Evidence can be stored in multiple ways:

1. **Inline in PR** — Paste compact format into PR description
2. **PR comment** — Post as a comment on the PR
3. **File in repo** — Save to `docs/verification/` (for auditable projects)
4. **CI artifact** — Generated by CI pipeline

For most projects, inline in PR is sufficient. File storage is for regulated or auditable contexts.

## Rules

1. **Never fabricate evidence** — Only report checks that were actually run
2. **Always list gaps** — What you didn't check matters
3. **Include actual command output** on failures — don't summarize errors
4. **Timestamp evidence** — dates matter for audit trails
5. **One bundle per verification run** — don't mix results from different runs
